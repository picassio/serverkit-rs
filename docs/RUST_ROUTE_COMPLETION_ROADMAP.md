# serverkit-rs Rust Route Completion Roadmap

Date: 2026-07-04  
Status: Planning artifact — **no fallback/stub destination**

## Objective

Implement every frontend-referenced ServerKit API route as a real Rust backend capability. The final state has:

- **Zero Python in the serving path**.
- **Zero fallback/stub route families counted as complete**.
- `routes/stubs.rs` removed.
- `routes/compat.rs` either removed or reduced to real, intentionally named adapters only.
- A route-parity test proving every frontend API call is implemented by Rust.
- Browser QA proving every page renders without 404/405/shape errors.

This roadmap treats an empty response as valid only when it is a real domain state, e.g. “no projects exist yet” from a persisted Projects subsystem. It is not acceptable for a handler to return empty JSON solely because the subsystem is missing.

## Non-negotiable implementation rules

1. **No fake success for mutating endpoints.** POST/PUT/PATCH/DELETE must persist state, call the underlying system, enqueue real work, or return a typed domain error.
2. **No silent no-op integrations.** External providers may be unconfigured, but the route must have provider configuration, validation, status, and real request code.
3. **No page-only contracts.** Implement the service-layer contract, not just enough JSON for a React component.
4. **All long-running work goes through the Rust job system.** Backups, scans, builds, deploys, DNS sync, provider operations, and fleet commands must expose real job status/history.
5. **Typed domain models first.** Add sqlx migrations and Rust structs; avoid unstructured `serde_json::Value` except at provider/plugin boundaries.
6. **Legacy Python is specification only.** It can be read to match semantics, but the runtime path stays Rust.
7. **Every route family has tests.** Unit tests for parsers/commands, integration tests for routes, and at least one browser or API sweep for the page/flow.

## Definition of done

A route family is complete only when all of these pass:

- Frontend API inventory lists the routes as implemented with method coverage.
- `cargo fmt --all`, `cargo clippy --workspace --all-targets -- -D warnings`, `cargo test --workspace` pass.
- Route integration tests cover auth, validation, empty state, happy path, and at least one failure path.
- Migrations are idempotent and included in release tarballs.
- The page using the route family has zero API 404/405 and zero console shape errors.
- Documentation explains installation/configuration and operational risks.
- Any provider-backed route has disabled/unconfigured behavior that returns an explicit domain status, not a stub.

## Phase 0 — Contract ledger and automated parity gate

**Goal:** Make missing routes mechanically visible before writing subsystem code.

Deliverables:

1. `backend-rs/tests/route_parity.rs`
   - Extract or consume a generated list of frontend API paths/methods.
   - Compare against a route manifest exported by the Rust backend.
   - Fail CI on any unclassified frontend route.
2. `docs/API_ROUTE_LEDGER.md`
   - One row per route: method, path, owning crate/module, implementation status, response model, frontend caller, tests.
3. `scripts/audit_frontend_routes.py`
   - Parses `frontend/src/services/api/*.js` and normalizes template strings.
   - Requires manual review for ambiguous paths generated by JS string concatenation.
4. `scripts/audit_backend_routes.py`
   - Reads a checked-in Rust route manifest. Do not rely only on regexing Axum code.
5. CI gate:
   - `implemented | planned | removed-from-frontend` are the only valid statuses.
   - `stub | compat-empty | unknown` fail the release gate.

Acceptance:

- Ledger covers all currently discovered frontend route families (~900 distinct paths before manual normalization).
- Existing `compat`/`stubs` routes are marked as temporary debt, not complete.

## Phase 1 — Core platform foundation

These are cross-cutting dependencies for later route families.

### 1.1 Jobs and queue core (`sk-jobs`, `sk-queue`)

Route families:

- `/jobs/*`
- `/queue/*`
- `/deployment-jobs/*`
- `/performance/jobs/*`

Build:

- SQLite tables: jobs, job_attempts, job_logs, queues, queue_messages, scheduled_jobs.
- Tokio worker runtime with concurrency limits, cancellation, retry/backoff, leases, and crash recovery.
- Job log streaming and durable status.
- Queue message receive/complete/fail/requeue semantics.
- Scheduled jobs backed by cron expressions.

Why first: builds, backups, scans, DNS sync, webhooks, provider actions, and fleet commands all need this.

### 1.2 Projects, environments, workspaces, permissions (`sk-projects`, `sk-workspaces`)

### 1.3 Notifications and telemetry (`sk-notifications`, `sk-telemetry`)

## Suggested release slicing

- **v0.2.0:** Phase 0 + Phase 1 jobs/projects/workspaces/notifications baseline. Stubs still exist but are barred from new route families.
- **v0.3.0:** Apps/builds/deploy/source connections/runtime adapters.
- **v0.4.0:** Databases/backups/files complete.
- **v0.5.0:** Web/domains/SSL/DNS/Cloudflare complete.
- **v0.6.0:** Security/firewall complete.
- **v0.7.0:** Fleet/cloud/remote server agent complete.
- **v0.8.0:** Built-in extensions complete.
- **v0.9.0:** Plugins/marketplace/agent-plugins complete.
- **v1.0.0:** Route ledger has zero `planned`, zero `stub`, zero `compat-empty`; `routes/stubs.rs` deleted; full browser/API sweep and upgrade path validated.
